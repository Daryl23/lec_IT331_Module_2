1. CROSS-PLATFORM APPLICATION DEVELOPMENT


1.1 Introduction to Cross-Platform Development
Cross-platform development is the practice of building software applications that can operate
across multiple operating systems and device types from a single, unified codebase.
Historically, organizations had to maintain entirely separate development teams and codebases
for each platform they wanted to support — a costly and time-consuming proposition. The rise
of modern cross-platform frameworks has dramatically changed this model.


Rather than writing native Swift for iOS and native Kotlin for Android (plus separate web and
desktop implementations), developers can now author one application that compiles or renders
appropriately on all target environments. This convergence has made cross-platform
development a first-choice strategy for most product teams.


Common Target Platforms:
  • Android – The dominant global mobile OS, requiring APK/AAB distribution through the
    Google Play Store.

    •    iOS – Apple's mobile platform, requiring App Store review and distribution.
    •    Web – Browser-based delivery, often via Progressive Web App (PWA) techniques.
    •    Desktop – Native desktop apps for Windows, macOS, and Linux via Electron or .NET
         MAUI.
    •    Embedded Systems – Lightweight runtimes for IoT and smart devices.


Business Benefits of Cross-Platform Development:
   • Reduced Development Cost: A single codebase eliminates duplicated effort across
      platform-specific teams.
   • Faster Time-to-Market: Features can be shipped simultaneously across all platforms in
      a single release cycle.
   • Improved Maintainability: Bug fixes, security patches, and feature additions need only
      be applied once.
   • Consistent User Experience: A unified design system ensures branding and
      interaction patterns remain coherent across devices.
   • Broader Market Reach: Targeting multiple platforms simultaneously increases the
      potential user base from day one.



1.2 Native versus Cross-Platform Development
The choice between native and cross-platform development is one of the most consequential
architectural decisions a product team makes. Both approaches have valid use cases; the right
choice depends on performance requirements, team skills, release cadence, and long-term
maintenance strategy.

1.4.2 React – Origin and Philosophy
    •    Built by Meta (Facebook), 2011: Created internally to solve the problem of building
         large-scale, dynamic UIs where data changes frequently. Open-sourced in 2013.
    •    Component-Based Architecture: React UIs are composed of self-contained, reusable
         components — functioning like Lego bricks. Each component manages its own
         structure, logic, and styling. Encourages modular design, making large codebases easier
         to navigate and maintain.

What is JSX?
JSX (JavaScript XML) is a syntax extension that allows HTML-like code to be written inside
JavaScript files. It is not valid HTML nor valid JavaScript on its own — a build tool (Babel or the
React compiler) transforms JSX into React.createElement() calls before the browser executes it.
JSX exists for developer ergonomics: it co-locates structure with logic and makes component
templates immediately readable.

Type Coercion — Why == is Dangerous in React:
JavaScript automatically converts one data type into another when using the loose equality
operator (==). For example: '5' == 5 evaluates to true because JS coerces the string to a
number. null == undefined is also true under loose equality. This implicit conversion is a source
of subtle, hard-to-debug errors. Always use === (strict equality) in React — it compares both
value and type without any coercion, eliminating this entire class of bug.

useState — Lifecycle Relation
    •    Mount: Sets the initial state value — e.g., const [count, setCount] = useState(0)

    •    Render: The current state value is read during render to decide what the UI should

    •    Update: Calling the setter function (setCount, setNotes, etc.) triggers a re-render. React
         re-runs the component function with the new state value.

useEffect — Lifecycle Relation
    •    Mount: Effect runs once when the component first appears (if dependency array is []).

    •    Unmount: The cleanup function returned from useEffect runs when the component is
         removed — stops timers, removes event listeners, cancels fetch requests.


  useEffect(() => {

       // Unmount: cleanup (remove listener, stop timer)

useRef — Lifecycle Relation
    •    Mount: The ref is initialized — e.g., const inputRef = useRef(null). The .current value is
         set to the initial argument.
    •    Render: The ref object stays identical across every render. Reading or writing .current is
         safe without causing re-renders.
    •    Unmount: Often used for cleanup — storing timer IDs, keeping the last known state

1.5 React Native – Building Native Mobile Applications

React Native is a framework for building native mobile applications using the React JS
paradigm. Unlike web-only React, React Native code does not run in a browser — it compiles to
real, platform-native code that runs directly on iOS and Android hardware. This is fundamentally
different from hybrid approaches such as Cordova or Ionic, which wrap a web view inside a
mobile shell.


What Does 'Native' Mean?
In the context of mobile development, 'native' means an application that is built using the
platform's own programming language and APIs — Swift/Objective-C for iOS, Kotlin/Java for
Android — and runs with full access to the device's hardware and OS capabilities. Native apps
deliver the best performance and deepest platform integration but require separate codebases
per platform.
    • React Native bridges this gap: you write JavaScript/TypeScript once, and React
        Native compiles it to call real native APIs and render real native components on each
        platform — not a web view in disguise.
    • Compare to Cordova/Ionic: those frameworks render HTML/CSS inside a WebView
        container. React Native skips the WebView entirely and communicates directly with
        native platform components.

Styling in React Native – StyleSheet API
React Native uses a JavaScript-based styling system rather than CSS. Styles are defined as
plain JavaScript objects and applied via the style prop. StyleSheet.create() validates and
optimizes styles at app startup.
    • No CSS Units: Numeric values represent density-independent pixels (dp). No px, em,
        rem, or % units — just numbers.
    • Flexbox Default: React Native uses Flexbox for all layout. The default flexDirection is
        column (vertical stacking) — the opposite of CSS default (row).
    • No Cascade: Styles do not cascade between components. Each component manages
        its own styles explicitly.
    • Platform-Specific Code: Use Platform.OS === 'ios' ? ... : ... or Platform.select({}) to
        apply different styles or logic per platform.

2. USER INTERFACE AND USER EXPERIENCE DESIGN


2.1 UI and UX Fundamentals
UI and UX are frequently conflated but represent distinct disciplines that must work in concert to
produce software that people love to use. The difference is not merely semantic — confusing
them leads to products that look polished but are frustrating to operate, or that are functionally
excellent but visually inconsistent and off-putting.


User Interface (UI) Design
UI design is concerned with the visual and interactive surface of an application. It asks: How
does each screen look? UI designers make decisions about:
    •    Visual Hierarchy: Guiding the user's eye to the most important element first through
         size, weight, color, and contrast.
    •    Layout and Spacing: Using grids, gutters, and whitespace to create organized,
         scannable interfaces.
    •    Color and Typography: Selecting a palette and typeface system that reflects brand
         identity while ensuring legibility.
    •    Component Design: Defining the visual and interactive behavior of buttons, forms,
         cards, modals, and navigation patterns.


User Experience (UX) Design
UX design takes a broader view, asking: How does the user feel throughout their entire journey?
A UX designer's toolkit includes:
   • User Research: Interviews, surveys, and usability tests that surface real user needs and
      pain points.
   • Information Architecture: Organizing content and navigation in a way that matches
      users' mental models.
   • Journey Mapping: Visualizing every touchpoint a user has with the product, from
      discovery through task completion.
   • Usability Heuristics: Applying Nielsen's 10 principles — such as error prevention,
      recognition over recall, and consistency — as a design quality checklist.
   • Prototyping and Testing: Building low-fidelity (paper) or high-fidelity (Figma)
      prototypes and iterating based on user feedback before a single line of code is written.



2.2 Responsive Design Principles
Responsive design is the practice of building interfaces that detect and adapt to the device's
screen size, orientation, and input method, providing an optimal experience across the full
spectrum of device types. A responsive application does not merely scale content up or down; it
thoughtfully reorganizes, shows, or hides elements based on the available viewport.

Core Responsive Design Techniques:
   • Fluid Grids: Column-based layouts where element widths are expressed in percentages
      rather than fixed pixels, allowing them to scale proportionally.
   • Flexible Images: Using max-width: 100% in CSS ensures images never overflow their

   • CSS Media Queries: @media (max-width: 768px) { ... } blocks apply different styles at
      defined breakpoints.
   • CSS Flexbox and Grid: Modern layout modules that natively support responsive
      arrangement — Flexbox for one-dimensional layouts, Grid for two-dimensional control.
   • Mobile-First Strategy: Begin by designing the mobile layout, then progressively add
      complexity for larger screens. This approach keeps the codebase lean and ensures core
      functionality works everywhere.
   • Viewport Meta Tag: <meta name="viewport" content="width=device-width, initial-
      scale=1.0"> — essential in every HTML document to prevent mobile browsers from
      rendering at desktop width.



2.3 Accessibility and Inclusive Design
Accessibility (often abbreviated a11y — 'a' + 11 letters + 'y') is the practice of designing software
so that it can be used by people with disabilities including visual, auditory, motor, and cognitive
impairments. The Web Content Accessibility Guidelines (WCAG) 2.1, published by the W3C,
are the international standard for digital accessibility, defining three conformance levels: A, AA,
and AAA.
In many jurisdictions, accessibility is a legal requirement. In the Philippines, the Magna Carta for
Disabled Persons (Republic Act 7277) and its amendments mandate accessible digital services
in public-facing systems.


Key Accessibility Considerations and Best Practices:
    •     Color Contrast: WCAG AA requires a minimum contrast ratio of 4.5:1 for normal text
          and 3:1 for large text. Use tools such as the WebAIM Contrast Checker to audit color
          pairs.



Batangas State University | College of Informatics and Computing Sciences                            Page 26
IT 331 – Application Development and Emerging Technologies                              Module 1.2



    •    Keyboard Navigation: Every interactive element (buttons, form fields, links, modals)
         must be reachable and operable using only the keyboard. Visible focus indicators must
         never be suppressed.
    •    Alternative Text: Images that convey information must carry descriptive alt attributes.
         Decorative images use alt="" to signal screen readers to skip them.
    •    ARIA Roles and Labels: WAI-ARIA attributes (role, aria-label, aria-describedby)
         communicate semantic meaning to screen reader software when native HTML elements
         are insufficient.
    •    Readable Typography: Body text should be at least 16px, line-height 1.5 or greater,
         and line length (measure) kept between 50–75 characters for optimal readability.
    •    Screen Reader Compatibility: Test with NVDA (Windows), VoiceOver (macOS/iOS),
         and TalkBack (Android) to ensure the application's content and controls are announced
         correctly.
    •    Error Identification: Form errors must be described in text, not only by color change,
         and should be associated with the relevant input using aria-describedby.


Business Case for Accessibility:
   • Broadens the user base — approximately 15% of the global population lives with some
      form of disability (WHO).
   • Reduces legal and compliance risk.
   • Improves overall usability for all users (the 'curb-cut effect': features built for accessibility
      often benefit everyone).
   • Enhances SEO — semantic HTML and descriptive alt text are also best practices for
      search engine indexing.



2.4 Design Systems and Reusable Components
A design system is a single source of truth for a product's visual language and interaction
patterns. It combines a component library, a style guide, and documented usage guidelines into
a shared resource that both designers and developers reference. Companies such as Google
(Material Design), Apple (Human Interface Guidelines), and IBM (Carbon Design) have
published comprehensive public design systems that serve as industry benchmarks.


Design System Layers:
   • Design Tokens: Named variables that encode the fundamental decisions: primary color
      (#2E75B6), base font size (16px), spacing unit (8px), border radius (4px). Tokens are
      the atoms of the system.
   • Component Library: A catalog of pre-built, documented UI components — buttons,
      inputs, cards, navigation bars, modals — with variants for different states (default, hover,
      disabled, error).
   • Layout Patterns: Approved grid systems, page templates, and content patterns that
      ensure structural consistency across screens.
   • Documentation and Governance: Guidelines on when and how to use each
      component, and a process for proposing additions or modifications.

3. STATE MANAGEMENT AND APPLICATION
ARCHITECTURE


3.1 State Management Concepts
State is any data whose change causes the application's interface to update. Managing state
correctly — knowing what data exists, where it lives, who can modify it, and how changes
propagate — is one of the most challenging and consequential aspects of application
development. Poorly managed state is the leading cause of hard-to-reproduce bugs and poor
application performance.


Types of Application State:
    •    UI State (Ephemeral State): Transient, widget-level data that does not need to persist
         across sessions. Examples: whether a dropdown is open, the current tab index, loading
         spinner visibility.
    •    Application State (Shared State): Data that multiple components or screens need to
         access. Examples: authenticated user profile, shopping cart contents, notification count.
    •    Server State: Data fetched from a remote API that must be cached, synchronized, and
         invalidated. Examples: product catalog, user feed, dashboard metrics.
    •    URL / Navigation State: The current route and query parameters. Deep-linking
         depends on keeping URL state synchronized with the application state.


Common State Management Approaches:
    •    setState / useState: Framework-native local state. Appropriate for UI state; does not
         scale to cross-component shared state.
    •    Provider / InheritedWidget (Flutter): Passes state down the widget tree using
         dependency injection. Low boilerplate; suitable for small-to-medium apps.
    •    BLoC (Flutter): Business Logic Component pattern that separates UI from logic via
         Streams. Highly testable; preferred for large Flutter apps.
    •    Redux / Zustand (React/React Native): Centralized immutable store with explicit
         actions and reducers. Redux offers powerful dev tools; Zustand is a lightweight modern
         alternative.
    •    MobX: Observable state that automatically re-renders components when observed data
         changes. Less boilerplate than Redux.



3.2 Client-Side and Server-Side Processing
A critical architectural decision in any application is where computation occurs — in the user's
browser/device (client side) or on a remote server. Modern applications are typically hybrid: the
frontend handles rendering and interaction, while the backend handles data integrity, security,
and heavy computation.


Client-Side Processing:


Batangas State University | College of Informatics and Computing Sciences               Page 29
IT 331 – Application Development and Emerging Technologies                                          Module 1.2



    •    Definition: Code executed on the user's device — in the browser (JavaScript/WASM) or
         in the mobile app runtime (Dart/JS/C#).
    •    Input validation and real-time form feedback.
    •    UI rendering, animations, and transitions.
    •    Client-side routing and navigation management.
    •    Caching and offline data access via Service Workers or local storage.
    •    Advantages: Low latency interactions; reduces server load; enables offline functionality.
    •    Disadvantages: Code is visible and reversible; cannot trust client-side validation alone
         for security.


Server-Side Processing:
    •    Definition: Code executed on a remote server controlled by the development team.
    •    Authentication and session management (JWT / OAuth 2.0).
    •    Authoritative business logic — pricing, tax calculations, eligibility rules.
    •    Database operations, complex queries, and data aggregation.
    •    File processing (image resize, document generation, ML inference).
    •    Advantages: Secure; trusted; can handle computationally expensive operations.
    •    Disadvantages: Network latency; requires backend infrastructure and DevOps.

3.4 Scalability Considerations
Scalability is the capacity of a system to handle growing amounts of work by adding resources
proportionally. A system that works well for 100 users may collapse under 10,000 if scalability
was not considered during design. There are two dimensions of scalability:
    •    Vertical Scaling (Scale Up): Adding more CPU, RAM, or storage to an existing server.
         Simple but has physical limits and creates a single point of failure.
    •    Horizontal Scaling (Scale Out): Adding more server instances behind a load balancer.
         Requires stateless application design (session state externalized to Redis or a
         database).


Architectural Decisions for Scalability:
   • Modular Architecture: Decompose the application into independent modules that can
       be scaled, deployed, and tested in isolation. The foundation for microservices.
   • API-First Development: Define and version API contracts before implementation.
       Enables frontend and backend teams to develop in parallel and supports third-party
       integrations.
   • Caching Strategy: Reduce database load by caching frequently-read data in Redis or
       Memcached. Apply cache invalidation carefully to avoid stale data.
   • Database Optimization: Index frequently-queried columns, normalize data
       appropriately, and consider read replicas for high-read-volume workloads.

    •    Cloud Deployment Readiness: Containerize with Docker; orchestrate with Kubernetes
         or OpenShift. Cloud-native apps can be auto-scaled based on CPU and memory
         metrics.
    •    Asynchronous Processing: Offload long-running tasks (email sending, report
         generation, file processing) to job queues (Bull, Celery, RabbitMQ) to keep the API
         responsive.

4.1 The Paradigm Shift: Systems Analysis vs. Screen Painting
Many student developers begin a project by asking: What database fields do we need? What
pages should we create? What should the dashboard look like? These questions are important
— but they should not come first.


When a system is designed from available data or from interface mockups alone, it often
becomes an information dump rather than a useful application. Users are forced to search for
information, interpret data themselves, and determine what actions to take — without any
support from the system itself.


Effective systems are designed differently. A successful Information System begins with
understanding the people who will use it, the problems they face, the decisions they must make,
and the processes they perform. The interface is simply a bridge between human goals and
system capabilities.

Q1 (Conceptual): What is the fundamental difference between Flutter and React Native in
terms of how they render UI?
Answer: Flutter renders its own pixels using the Skia/Impeller GPU engine, bypassing the
platform's native widget system entirely. React Native communicates with the platform's native
UI components via a JavaScript bridge (or the newer JSI), resulting in truly native widgets that
inherit the platform's look and feel. Flutter produces pixel-identical output across platforms;
React Native inherits each platform's visual conventions.


Q2 (Scenario): A team building a university portal has a JavaScript-centric React
background. They need to target iOS, Android, and web simultaneously. What is the
most appropriate cross-platform strategy?
Answer: React Native for iOS/Android combined with a responsive React web application
shares the most code — business logic, API integration, and many component patterns —
across all targets. If native device features are not required, a PWA-first approach using Next.js
with mobile-responsive design may suffice. The team's existing JavaScript expertise makes
both options lower-risk than adopting Flutter (Dart) or .NET MAUI (C#).


Q3 (React): A useEffect runs every time the component renders, causing an infinite loop.
What is the most likely cause, and how do you fix it?
Answer: The dependency array has been omitted, which causes the effect to run after every
render. If the effect also updates state, it triggers another render — creating the loop. Fix: add a
dependency array. Use [] if the effect should run once on mount, or [specificValue] if it should
re-run only when that value changes. Only omit the dependency array when the effect genuinely
must run on every render, which is rarely correct.


Q4 (9-Level Audit): A student presents a dashboard with five widgets: Total Users,
Monthly Revenue chart, Revenue by Region pie chart, Revenue by Region table, and a
'Latest Transaction' list. Apply Levels 4, 6, and 8 of the 9-Level Audit. Which components
are justified and which should be removed?
Answer: Level 4 (Core Inquiry) — each component needs an articulable question. 'Total Users'
answers 'How many users do we have?' (justified if that is a primary question for this role).
Monthly Revenue chart answers 'Is revenue trending up or down?' (justified). Level 6 (Attention
Density) and Level 8 (Redundancy) — Revenue by Region as both a pie chart and a table on
the same screen is redundant: they answer the same question with the same data.
Recommendation: keep the pie chart for proportional comparison (faster to read); move the
table to a drill-down view accessible on click. The 'Latest Transaction' list is justified only if this
role's primary task involves monitoring recent activity; otherwise it is optional and should be
moved to a secondary panel.


Q5 (Accessibility): A button in your UI reads 'Click here'. Why is this a WCAG failure, and
how would you fix it?
Answer: 'Click here' is non-descriptive — a screen reader user navigating by button will hear
'Click here' with no context about the action's purpose. WCAG 2.4.6 and 2.4.9 require link and
button text to be self-describing. Fix: replace with a label that describes the action and target:

'Submit Grade Appeal', 'Download Transcript', or 'Register for CS 201'. If an icon-only button
must be used, add an aria-label attribute with the descriptive text.
